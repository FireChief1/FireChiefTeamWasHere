# JavaScript / Node.js Style Guide

Standards for standalone Node.js and JavaScript modules generated by the team.

## Modules and exports

Use ES modules (`import` / `export`) by default. Export the public API so it can
be imported and tested:

```js
export function parse(input) { /* ... */ }
export class Stack { /* ... */ }
```

Keep one cohesive responsibility per file. Use `.mjs` only when a project mixes
module systems; prefer plain `.js` with `"type": "module"` semantics otherwise.

## Naming and structure

Use `camelCase` for variables and functions, `PascalCase` for classes, and
`UPPER_SNAKE_CASE` for constants. Prefer small, pure functions. Avoid deep
nesting; return early.

## Language features

Prefer `const`, then `let`; never use `var`. Use strict equality (`===`).
Use template literals over string concatenation. Use optional chaining (`?.`)
and nullish coalescing (`??`) where they clarify intent.

## Errors and async

Throw `Error` (or a subclass) with a clear message; do not throw strings.
Always `await` promises or return them; never leave unhandled rejections.
Do not block the event loop with long synchronous work.

## Safety

Never use `eval` or the `Function` constructor on untrusted input. Validate and
sanitize external input. Do not log secrets.

## Testing expectations

Code is verified with Node's built-in test runner (`node:test`) and assertions.
Write code that is import-friendly and deterministic so tests can call it
directly. Avoid side effects at import time.
